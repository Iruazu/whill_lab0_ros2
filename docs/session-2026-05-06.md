# Session log: 2026-05-06 — M2 wrap-up on real CR2

First end-to-end bring-up of the WHILL Model CR2 on the ROS 2 humble stack.
This log preserves the *investigation path* (including dead ends) so the
next person doesn't have to retrace the same misdiagnoses.

## Setting

- Lab PC moved into the WHILL room and connected to `uu-w` (different subnet
  from the one the campus HTTP proxy resolves on).
- Existing system-wide `HTTP_PROXY` / `HTTPS_PROXY` env vars were `unset` in
  the launching shell so they didn't interfere with USB / ROS work.
- Chair was powered on with the USB already plugged in (LCD lit, showing the
  number `93`, LED red at startup).

## Symptom

`ros2 launch whill_bringup whill_launch.py` opened `/dev/ttyUSB0` cleanly.
The whill node stayed alive (multi-thread, no errors in logs), held fd 16
on `/dev/ttyUSB0`, and `/whill/states/model_cr2` appeared in `ros2 topic
list`. But:

- `ros2 topic hz /whill/states/model_cr2` produced no rate samples
- `ros2 topic echo /whill/states/model_cr2 --once` timed out
- `ReceiveDataset1()` in the driver was returning `< 1` every tick — i.e.
  no bytes were arriving from the chair

So the host-side was healthy, the chair-side was sending nothing.

## Misdiagnoses (kept here so we don't loop on them again)

1. **"Red LED + numeric display = error code"** — false. Acted on this
   without consulting the spec, then reversed when the operator pointed out
   that `93` is just the battery percentage and a brief red LED is part of
   the normal boot sequence. **Lesson: read the protocol PDF before guessing
   based on generic device conventions.**
2. **"USB plugged in before chair power-on, so re-enumerate"** — replugged
   the USB after a power cycle. `/dev/ttyUSB0` came back, but the symptom
   was unchanged. Order of plug/power was *not* the cause.
3. **"PL2303 chip mismatch / cable wiring suspect"** — spec says CR2
   communication is RS232C ±12V on a D-sub 9pin (Pin2 TXD, Pin3 RXD, Pin5
   GND), and this lab uses a custom 3-wire harness. Suspected TX/RX swap or
   level mismatch. **False alarm**: the harness works fine; bytes were
   leaving the host (8 bytes wrote successfully via raw `os.write`), and
   bytes did come back once the chair was woken. The cable is not the
   problem.

## Root cause

A freshly-powered Model CR2 does **not** start streaming Dataset1 frames
just because `StartSendingData` arrived — its serial communication
subsystem stays dormant until the host sends `SetPower(ON)`. This is
documented in the WHILL Control System Protocol Specification, section 5
("WHILL response data"), but the upstream `whill-labs/ros2_whill`
`Initialize()` only sends `StartSendingData`, never `SetPower`.

Discovered by stripping the driver out of the loop and writing the
`SetPower(ON)` byte sequence (`AF 03 02 01 AF`) directly to
`/dev/ttyUSB0` at 38400 8N2 from a small Python probe. The chair
immediately responded with the documented response packet
(`AF 02 52 FF`) and then started sending 33-byte Dataset1 frames at the
configured 400 ms interval.

**Side-effect found while testing:** after `SetPower(ON)`, the chair flips
into host-control mode, which silently disables the body joystick. A
follow-up `SetJoystick` with `U0=0x01` (kUserControlEnable) hands control
back to the user.

## Fix

Patch in `Iruazu/ros2_whill` PR #1 — adds, in `WhillNode::Initialize()`
just before `StartSendingData`:

```cpp
whill_->SetPowerOn();
usleep(10000);
whill_->SetPowerOn();
usleep(2000);
whill_->SendSetJoystickCommandWithLocal();
usleep(2000);
```

The double `SetPowerOn()` mirrors the existing retry pattern in
`OnSetPowerSrv`. ROS-side control paths (`SetVelocityCommand` for
`/whill/controller/cmd_vel`, `SetJoystickCommand` for
`/whill/controller/joy`) still flip the chair to host-control on demand,
so this change does not regress ROS-driven motion.

## Lessons

- **Read the spec before extrapolating from device-generic intuition.** The
  protocol PDF in
  [`WHILL/whill_control_system_protocol_specification`](https://github.com/WHILL/whill_control_system_protocol_specification)
  is the single source of truth for byte-level behavior. The first hour of
  this session would not have been spent on cable theories had I started
  there.
- **Trust the operator's local readings** — when the operator said "93 is
  battery", that contradicted my "red LED + number = fault" framing. They
  were right; I should have updated faster.
- **Driver silence is not the same as driver failure.** The whill node
  log was clean from start to finish — the issue was that `ReceiveDataset1`
  silently returns `< 1` when no bytes arrive, with no warning. A future
  improvement could log a warning if zero bytes have arrived for, say, the
  first 2 s after startup.
