## What M5-e is trying to fix

User observation on this branch, with the chair seated still on the
m5e/velodyne-self-filter branch:

> 進行方向に対して円弧のような壁が発生しております。
> lidarのマウント材所以なので ロボットの動作に伴い 壁も前方に
> 更新され連なっているように見える

Diagnosis: the LiDAR mount frame sits at a roughly fixed radial
distance from the velodyne origin, just below the LiDAR plane. The
existing filters can't reach it:

- FAST-LIO's `blind: 0.5` cuts only points within 0.5 m radial.
- `fov_degree: 180` cuts the back hemisphere but the mount is
  forward of the LiDAR.

So the mount returns survive into the front half-disc of every
scan. FAST-LIO bakes them into the world map; as the chair drives
forward, each new scan stamps a fresh arc at the chair's current
position. The arcs blend into a continuous tunnel ("壁が前方に
更新され連なって"), which is the phantom corridor that motivated
the whole filter idea.

## What's already in place (commit 6049b35)

- New `scripts/velodyne_self_filter.py` in whill_sensors_bringup.
  Cylindrical self-exclusion: drops a point if
  `sqrt(x²+y²) < self_radius` AND `self_z_min < z < self_z_max` in
  the velodyne frame. Defaults to a no-op (`self_radius: 0.0`) so
  the filter cannot break FAST-LIO until the geometry is dialed in.
  Logs a kept-ratio every 100 messages.

- `whill_sensors_bringup/launch/sensors_launch.py` wires the filter
  alongside the velodyne / realsense / IMU drivers with the no-op
  defaults.

- `whill_localization/config/velodyne_whill.yaml` keeps `lid_topic:
  /velodyne_points` (raw) so FAST-LIO is unaffected. We'll flip
  this to `/velodyne_points_filtered` after the geometry is dialed.

## False starts and what we learned from them

1. **Naive z-only filter** (`keep z >= -0.10 m`) — committed and
   immediately reverted (`6049b35`). The filter dropped every
   floor return, FAST-LIO ran out of effective points in seconds,
   base_link drifted out of the costmap, pcl::VoxelGrid hit the
   int32 overflow. Lesson written into the commit body: VLP-16
   needs both upper and lower rings for scan-to-map registration.
   The mount lives in a small region around the LiDAR, not in an
   entire z half-plane.

2. **FAST-LIO subscription QoS** is `rclcpp::SensorDataQoS()`
   (verified in `src/third_party/FAST_LIO/src/laserMapping.cpp:927`),
   so a BEST_EFFORT publisher from the filter matches. QoS was not
   the cause — the empty kept-set was.

## Next-session entry point (M5-e measurement)

Resume immediately after reboot with:

```bash
cd ~/whill_lab0_ros2
git checkout m5e/velodyne-self-filter && git pull --ff-only
source install/setup.bash
ros2 launch whill_navigation nav_launch.py    # T1
ros2 launch whill_bringup whill_launch.py     # T2
```

Then in RViz, with chair seated still:

1. **Fixed Frame → `velodyne`** (NOT `map`). Switching to the LiDAR's
   own frame freezes the arc in space so it can be measured; in the
   `map` frame the arcs trail behind the moving chair and the geometry
   is hard to read.

2. **Two PointCloud2 displays**:
   - `/velodyne_points` (raw, reliability Best Effort, size 0.02)
   - `/velodyne_points_filtered` (currently a pass-through; will
     visibly differ from raw once `self_radius > 0`).
   Add an `Axes` display to see the velodyne origin.

3. **Use the Measure tool** (toolbar / `m`):
   - Click 1: velodyne origin (frame axis center).
   - Click 2: a dense point on the front arc.
   - Read the distance. Report it back as `self_radius_observed`.

4. **Estimate z range** by tilting RViz to a side view (or rotating
   manually) and reading the top / bottom of the arc strip. Report
   as `self_z_min_observed`, `self_z_max_observed`.

5. With those three numbers, set them in
   `src/whill_sensors_bringup/launch/sensors_launch.py` (the
   velodyne_self_filter Node block) — bump `self_radius` to slightly
   larger than the observed value so the filter actually swallows
   the arc. Defaults to start from if measurement is skipped:
   `self_radius: 1.0`, `self_z_min: -0.3`, `self_z_max: 0.1`.

6. Rebuild + relaunch:

   ```bash
   colcon build --packages-select whill_sensors_bringup --symlink-install
   source install/setup.bash
   ros2 launch whill_navigation nav_launch.py
   ```

   Then in RViz toggle the two PointCloud2 displays on/off and confirm
   the arc disappears in `/velodyne_points_filtered` while the rest of
   the scene survives. The filter logs a kept-ratio every 100 msgs;
   anything in the 80–95% range is healthy, well under 50% means the
   cylinder is eating too much.

7. Once the visual diff looks right, flip
   `src/whill_localization/config/velodyne_whill.yaml`:

       lid_topic: "/velodyne_points_filtered"

   Rebuild whill_localization, relaunch, send the same 1.6 m goal as
   M5-d, and look at FAST-LIO's accumulated map: the phantom corridor
   should be gone. If FAST-LIO regresses to "No Effective Points",
   shrink `self_radius` until registration recovers.

8. If 1–6 succeed, the M5-c-era hacks become removable:
   - `pcd_to_occupancy_grid.py --clear-radius` can stay default but
     is no longer load-bearing.
   - `FollowPath.use_collision_detection` can flip back to `true`
     once a fresh lab.pcd is captured without people.
