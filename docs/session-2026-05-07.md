## Hardware findings

### USB topology held across the chair install

Before the move the RealSense and the RTL8153 USB-Ethernet adapter
were on Bus 002 of the host's xHCI controller; after the chair install
they were on Bus 004. WHILL stayed on Bus 003 but its port number
changed too. `/dev/whill` and `/dev/imu` resolved correctly on first
launch — exactly the case the VID:PID-based udev rule was written for.
This is the data point that justifies the udev approach over `KERNELS=`
USB-path matching.

### RealSense is D435, not D455 as the handover note said

`dmesg` reports `Intel(R) RealSense(TM) Depth Camera 435` and the
device's product ID is `0x0B07` (Intel's official mapping has
`0b07 = D435`, `0b5c = D455`). The user confirmed by physical
side-label. Driver-wise this changes nothing — the same `realsense-ros`
release supports both — but several places in the M3 docs needed the
correction.

### Velodyne network configuration

VLP-16 factory IP is `192.168.1.201`; host gets `192.168.1.100/24` on
the USB-Ethernet adapter via netplan. Ubuntu Desktop ships a
`renderer: NetworkManager` netplan, and netplan rejects mixing
renderers across `/etc/netplan/*.yaml`, so the tracked template uses
`NetworkManager` too (with a comment about how to flip it for
networkd-only hosts). Verified end-to-end on the chair: link UP, ARP
resolves the LiDAR's MAC `60:76:88:10:6b:6b`, ping 0% loss,
`tcpdump udp port 2368` shows the LiDAR streaming `length 1206`
packets to `255.255.255.255:2368` at full rate.

## Software findings

### IMU is a `LifecycleNode` and the configure-then-activate transition is racy

The upstream `rt_usb_9axisimu_driver` doesn't publish anything until
it has been transitioned `configure → activate`. The first version of
`imu_launch.py` did this via `EmitEvent + OnStateTransition`, which
fires the `activate` event the moment `configure` returns — about 22
ms later in practice.

That worked on the bench but broke after the chair mount. The driver's
`on_activate()` callback calls `readSensorData()` synchronously, and
if the kernel serial buffer hasn't yet collected a full ASCII frame
from the device the read returns `FAILURE` and the node bounces to
`errorprocessing`.

Fix in `69c3980`: insert a `TimerAction(period=1.5)` between the
`OnStateTransition(goal_state='inactive')` handler and the
`Transition.TRANSITION_ACTIVATE` `EmitEvent`. After the change, the
log reads:

```
on_configure() is called.
reached 'inactive', waiting 1.5 s before activate to let the serial
  port buffer a frame.
on_activate() is called.
ros2 lifecycle get /rt_usb_9axisimu_driver  →  active [3]
```

The same pattern likely applies to other USB-serial / CDC-ACM
lifecycle drivers — keep it in mind for M4 if FAST-LIO needs an
extra preprocessing lifecycle node.

### Other small surprises

- **`ros2 topic hz` in humble does not auto-detect best-effort QoS.**
  The Velodyne driver publishes with sensor-data QoS, so `hz` returns
  no samples; `ros2 topic echo --once` works, or write a Python
  subscriber with matching QoS.
- **RealSense topic prefix is `/camera/camera/...`.** `rs_launch.py`
  uses parent ns `camera` + node name `camera`, so the topics are
  doubly prefixed. Downstream M4 / M5 launches need to remap or accept
  this prefix instead of the bare `/camera/`.

## Lessons

- **VID:PID-based udev was the right call.** USB ports moved when the
  laptop was physically remounted, but the symlinks held. If the rule
  had used `KERNELS=` USB-path matching, the chair install would have
  silently broken `/dev/imu`.
- **Don't trust handover notes blindly.** "8086:0b07 is a D455" —
  cited in the original migration plan — was wrong. `dmesg` and the
  Intel ID table are authoritative; a 30-second check would have
  caught this earlier.
- **Lifecycle nodes need backoff between transitions.** Fast
  `configure → activate` chains race with the underlying device's
  data-ready window. A `TimerAction` is the cheapest insurance.
- **Verify "stationary" data programmatically, not by feel.** The
  user's intuition that the first static bag was contaminated was
  correct, and the IMU stats backed it up. `scripts/check_static.py`
  is now there to make that check a one-liner for any future bag.
