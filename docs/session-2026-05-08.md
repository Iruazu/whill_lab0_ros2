## The identity-extrinsic detour

The 2026-05-07 commit had reverted the LiDAR↔IMU extrinsic to identity
because, on a single replay run that day, the noetic-inherited values
had diverged within ~30 s while identity gave a bounded trajectory.
That looked like clean evidence that the noetic extrinsic was stale.

It was not. Today's reproducibility study showed:

- The same `m3_chair_motion` bag, replayed twice in succession with
  identity extrinsics, produced two completely different divergent
  trajectories (end pose 4× and 7× kilometres off, in different
  directions). Yesterday's bounded result was a multi-threading lucky
  outcome.
- The user confirmed in chat that the chair's physical mount has not
  been touched since the noetic era. The noetic translation
  `[0.104, 0.412, 0.324]` matches exactly: LiDAR on the chair's left
  (+0.412 m), under the seat cushion ~30 cm below the IMU (+0.324 m
  in z), small forward offset (+0.104 m).

So the right move was always to keep the noetic values. With them
restored and `gyr_cov` loosened, the same bag gave a reproducible
bounded trajectory on the cleanest of the three new runs.

The correction is in `bc52e8b`; both `whill_localization/README.md`
and `docs/m3-bench-data/README.md` document the journey so the next
person doesn't repeat the detour.

## Three-run reproducibility study

Same simple route (corridor + a turn into a chair/desk room and back
near the start), same WHILL speed mode 2, three back-to-back 60 s
recordings with the chair returned to the start position between
runs. Each run got a fresh FAST-LIO instance.

| Run | Capture quality | Replay outcome |
|-----|-----------------|----------------|
| run1 | static window mistimed against user's stopwatch — recorded data started before the chair was actually still | diverges from t=9 s onward; iESKF never gets a clean IMU bias estimate |
| **run2** | **clean static + clean drive** | **bounded; 40 m path; 14 m max displacement; 18 % loop-closure error** |
| run3 | a pedestrian crossed the front 180° FOV in the second half | diverges where the pedestrian appears; moving features in the world map break registration |

Replaying `run2` a second time produced a similar shape (50 m path,
22 % loop-closure) with end pose ~10 m off the first replay's end.
That residual variation is FAST-LIO's multi-threaded kdtree updates
being non-deterministic; the trajectory shape is reproducible.

### Lessons

- **Capture quality dominates.** With this config a clean 60 s drive
  is fine; a sloppy static window or an unannounced pedestrian breaks
  localization irrecoverably. Future drives should announce
  themselves and run during quiet windows.
- **The "noetic extrinsic is stale" instinct was wrong** — when a
  physical mount hasn't changed, the inherited calibration is still
  valid. Always confirm with the operator before assuming
  recalibration is needed.
- **Multi-threading in FAST-LIO is non-deterministic.** Same bag,
  same binary, same config, two consecutive replays differ by ~10 m
  end-pose on a 50 m path. iESKF + kdtree concurrency is the likely
  culprit. Live with it for now (M5 doesn't require single-pass
  determinism), file an upstream issue later if needed.
- **Live `/Odometry` was sparse (~1.5 Hz)** because `fastlio_mapping`
  shared the host with RViz, three sensor drivers, the static TF
  publishers, and `ros2 bag record` writing 3.3 GiB to disk. Offline
  replay against just `fast_lio_launch.py` recovers the full ~7 Hz.
  This affects update rate, not pose correctness — but for M5 / Nav2
  the controller may want to subscribe to live `/Odometry` directly,
  so we may need to either reduce host load (skip RViz, skip the
  RealSense streams when they aren't needed) or run FAST-LIO on a
  separate host.

## Two side bugs found and fixed in `bc52e8b`

- `localization_launch.py` was including `fast_lio_launch.py` via
  `IncludeLaunchDescription`. The inner launch had
  `parameters=[LaunchConfiguration('config_file'), …]`. When wrapped,
  the substitution silently resolved to an empty string and the
  params file was dropped from the `fastlio_mapping` command line.
  FAST-LIO then loaded its upstream defaults — `lidar_type=AVIA`,
  Livox topic names — and SIGSEGV-ed on the first Velodyne
  PointCloud2. Standalone `ros2 launch whill_localization
  fast_lio_launch.py` was unaffected because the LaunchConfiguration
  resolved fine in the top-level scope. Workaround: hardcode the
  config path; only use a LaunchArgument for things that genuinely
  need to vary at launch time.
- `cube_side_length` was at upstream's 1000.0 with `filter_size_map`
  at 0.5, which makes pcl::VoxelGrid's int32 indexing overflow
  (1000³ / 0.5³ = 8 × 10⁹ > 2³¹). Symptom is spammy
  `Leaf size is too small for the input dataset` warnings followed
  by a steady-state "No Effective Points!" stall. Reduced to 200.0.

## Tomorrow's first steps (M5 / Nav2)

M4 is merged. The chair-driven `run2` replay is the M4 deliverable.
The next milestone is M5 (autonomous navigation via Nav2). Offline
work first; chair time can be saved for the live validation phase.

```bash

git checkout -b m5/navigation
```

Suggested rough order:

1. **Wire FAST-LIO output into Nav2's expected TF tree.** FAST-LIO
   publishes `camera_init -> body`. Nav2 expects `map -> odom ->
   base_link`. Either remap FAST-LIO's frames or insert a small bridge
   node (`tf2_ros::StaticTransformBroadcaster` or a custom relay) to
   produce `map -> odom` from `camera_init -> body` and let
   `whill_sensors_bringup`'s static TF carry `odom -> base_link`.
2. **Build a saved map.** Flip `pcd_save.pcd_save_en` to true, drive a
   slow loop covering the area Nav2 will operate in, save the PCD.
   Convert to a 2D occupancy grid (e.g. `pcd_to_pgm`, or use
   `octomap_server`) so AMCL / Nav2's costmap can consume it.
3. **Stand up Nav2 bringup.** `nav2_bringup` template launch with our
   own `nav2_params.yaml`. Local + global costmap on the saved map +
   live `/Odometry`. Use the standard `bt_navigator` and
   `controller_server` for now.
4. **Live drive test.** Send a `geometry_msgs/PoseStamped` goal in
   RViz, watch the chair plan a path and follow it via
   `cmd_vel` -> `whill_driver`. Tune costmap inflation, planner
   cadence, and `controller_server` speed limits to suit the chair's
   drive characteristics (Mode 2 limit ~).

Closing condition for M5: `ros2 launch whill_navigation
nav_launch.py`, drop a 5 m goal in RViz, the chair plans and reaches
it autonomously without colliding with anything in the costmap.

### M5-a — TF bridge (commit `3de4ea1`)

New `whill_navigation` package. `tf_bridge_launch.py` adds two
identity `static_transform_publisher` nodes — `map → camera_init`
and `body → base_link` — that turn FAST-LIO's `camera_init → body`
into a Nav2-shaped `map → ... → base_link` chain when combined with
the M3 static TFs. Verified offline by replaying the run2 chair bag
against `fast_lio_launch.py + tf_bridge_launch.py` and capturing
`tf2_tools view_frames`. The snapshot
(`docs/m3-bench-data/frames-m5a-2026-05-08.pdf`) shows a single
rooted tree from `map` down through every sensor frame including
the RealSense `camera_*_optical_frame` leaves.

### M5-b — Saved PCD (commit `6d8b299`)

First PCD checkpoint at `docs/m5-maps/lab.pcd` (256,478 points,
8.2 MB). Authored by replaying run2 against fast_lio (with the new
config flags `pcd_save.pcd_save_en: true` and
`publish.map_en: true`) and calling the `/map_save` service over
std_srvs/Trigger when the bag finished.

Two upstream FAST-LIO quirks got documented in the process:

1. Setting `pcd_save_en: true` alone does **not** save anything.
   The in-loop save path that would have populated `pcl_wait_save`
   from `publish_frame_world` is commented out in
   `src/laserMapping.cpp`. The shutdown handler therefore has no
   buffered points to write.
2. `pcl_wait_pub` (the buffer the `/map_save` service writes) is
   only fed by `publish_map()`, which only runs when
   `publish.map_en: true`. That flag was off by default. Flipping
   it on is what makes the service have data.

Caveat for M5-c: the saved PCD includes scattered points from the
FAST-LIO drift segments. Bounding box hits X ∈ [-352, +71],
Y ∈ [-69, +323], Z ∈ [-10, +90] m even though run2's bounded
trajectory stayed within ~15 m of origin. The PCD → 2D occupancy
grid conversion in M5-c will need to clip aggressively (XY box,
Z slice for chair-relevant heights) before `nav2_map_server` can
consume it.

### Process pollution episode

Around 15:00 the M5-b orchestration tripped over leftover
`ros2 launch` processes from earlier turns of the conversation —
`/tmp/m5b_pcd_save.sh` had been started multiple times in different
Bash invocations and the kills only ever hit the foreground
processes, leaving orphaned static_transform_publisher and the wrapped
`ros2 launch` python processes alive. Symptom: each new launch
thought ports / TFs were already taken.

The recovery was straightforward — `pkill -9 -f` against every
process pattern with `grep -v antigravity` to spare the editor
language server. Worth remembering for future M5/M6 work where
several launch files run side by side: prefer `kill -9` on the
launch root pid rather than counting on `pkill -INT` to cascade
through `ros2 launch`'s child supervision.
